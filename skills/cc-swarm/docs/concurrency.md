# 并发调度 (concurrency)

控制同时活跃的 session 数量，防止 API 429 限流。

## 问题

Claude Code API 有请求频率限制。当多个 session 同时活跃时，它们会竞争同一个 API 配额，导致 429 错误。429 后 session 会重试，但重试本身也消耗配额，形成恶性循环。

## 策略

### 并发上限

- **同时活跃 session 不超过 2-3 个**
- "活跃"定义：正在执行（写代码、跑命令、读文件），不包括 idle 在 prompt 的

### 判断谁在活跃

从 capture 输出识别：
- `esc to interrupt` → 执行中（活跃）
- `Running...` / `Reading...` / `Writing...` → 执行中（活跃）
- 空 prompt `❯` + 最近有完成时间 → 空闲
- `429` / `Rate limited` → 被限流（算半活跃，在重试队列里）

### 调度决策

```
统计当前活跃数
  ├── < 2  → 可以给空闲 session 发新指令
  ├── = 2  → 不发新指令，只做审批
  └── > 2  → 不发新指令，考虑等一个完成再行动
```

### 发指令的间隔

- 给不同 session 发指令之间间隔 **20-30 秒**
- 不要在一个巡检循环里给所有空闲 session 都发
- 让先发的 session 跑起来（消耗它的 API 配额周期），再发下一个

### 429 处理

当发现 session 遇到 429：
- **不要重发指令** — Claude Code 自己会重试
- **不要给其他 session 发新指令** — 减少整体负载
- **拉长巡检间隔** — 从 30s 拉到 60s
- **等 429 的 session 恢复后再正常调度**

## 优先级

当并发位有限时，哪些 session 优先运行？

1. **已经在执行中的** — 不打断，让它跑完
2. **等审批的** — 卡住的优先解除（审批本身不消耗 API 配额）
3. **任务更关键的** — 阻塞其他工作的优先
4. **快完成的** — 快做完的优先，释放位置

## 反模式

- ❌ 同时给 5+ session 发指令
- ❌ 用同样的模板发，导致同时触发相似的 API 请求
- ❌ 429 了还继续发新指令
- ❌ 用 sleep 0.5 快速循环轮询（自身也消耗 API）
