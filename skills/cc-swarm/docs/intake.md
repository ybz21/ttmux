# 接需求 + 建需求 (intake & spec)

cc-swarm 拿到的 `$ARGUMENTS` 通常是一句模糊的目标。第一件事**不是开会话**，
而是把它变成一份清晰、可拆分、可验收的**需求规格**。

> 原则：宁可多花两分钟问清楚，也不要拿模糊目标去开五个会话各跑偏。

## Step 1: 理解目标

读 `$ARGUMENTS`，再快速侦察项目本身（不要瞎猜）：

```bash
ttmux ls                 # 已有哪些会话？是否已有人在做相关的事
```

用 Read/Grep/Glob 或 Explore 子 agent 摸清：
- 项目用什么技术栈？（看 package.json / pyproject.toml / go.mod …）
- 已有的代码结构、约定、测试框架是什么？
- 目标涉及的模块现在长什么样？

**目标是知道「这事在这个项目里具体意味着什么」，而不是泛泛而谈。**

## Step 2: 判断要不要追问

只在**真正影响拆分或实现方向**的地方问，别问能自己查到或有合理默认的。

| 该问 | 不该问 |
|------|--------|
| 范围边界（"OAuth 要支持哪些 provider？"） | 能从代码看出来的技术栈 |
| 互斥的方向选择（"新建服务还是改现有的？"） | 有明显默认的小决定 |
| 验收标准不明（"算完成的标准是什么？"） | 实现细节（让子会话自己定） |
| 优先级/截止（"先要能跑还是要完整？"） | — |

如果要问，**一次把关键问题集中问完**（用 AskUserQuestion 给选项），不要挤牙膏。
如果目标已经足够清楚，跳过追问直接建规格。

## Step 3: 写需求规格

把目标固化成一份 markdown 规格，存到蜂群记忆里：

```
~/.local/share/ttmux/cc-swarm/_spec.md
```

格式：

```markdown
# 需求：<一句话目标>

## 背景
为什么做这个，解决什么问题（1-2 句）。

## 范围
### 要做
- 明确包含的功能点

### 不做（out of scope）
- 明确排除的，避免子会话过度发挥

## 技术约束
- 技术栈 / 框架 / 风格约定（从项目侦察得出）
- 必须复用的现有模块、不能破坏的接口

## 验收标准
- [ ] 可观察、可验证的完成条件（不是"做好了"，而是"X 能跑出 Y"）
- [ ] 测试要求
- [ ] 集成要求（前后端对接、迁移脚本等）

## 风险 / 未决
- 已知的难点、依赖外部的东西
```

## Step 4: 复述确认（可选但推荐）

把规格的**范围 + 验收标准**用一两句话向用户复述一遍，确认理解一致后再进入拆任务。
如果是 `plan` 模式，规格 + 拆分结果一起给用户看，等确认。

## 反模式

- ❌ 拿到 "做个用户系统" 直接开 5 个会话 —— 没人知道边界，结果东拼西凑。
- ❌ 把所有细节都问用户 —— 你是来减负的，不是来做问卷的。
- ❌ 规格只存在脑子里 —— 必须落到 `_spec.md`，拆任务和验收都要回头对照它。
