# 集成验收 (integrate)

所有子任务完成后，cc-swarm 要把成果**收拢、对接、对照需求验收**，再向用户汇报。
不做这一步，蜂群就只是开了一堆会话，没人确认目标真的达成了。

## 触发时机

- 所有 `_tasks.md` 里的子任务都标记为 done；或
- 用户显式 `/cc-swarm integrate`。

## 流程

### Step 1: 收拢成果

逐个子会话确认产出：

```bash
ttmux capture cc-<name> --lines 60        # 交互式会话的最终总结
ttmux agent collect <group> --json        # 无人值守会话的产出
```

从每个会话提取：改了哪些文件、对外接口/契约是什么、有没有遗留 TODO。
记到蜂群记忆，更新各会话记忆文件的「产出」段。

### Step 2: 检查对接点

跨会话的接口是集成最容易出问题的地方，重点查：

- **契约一致吗？** 后端实际返回的字段 == 前端实际消费的字段？（拆任务时定的契约，实现时可能漂了）
- **命名/类型对齐吗？** snake_case vs camelCase、id 是 int 还是 string。
- **依赖装齐了吗？** 新加的包都进了 lockfile / requirements？
- **迁移/配置到位吗？** DB 迁移跑了吗？新的 env 变量文档化了吗？

发现对接断裂 → 给相关会话发具体的修正指令（哪个字段对不上、改哪边），不要笼统说"对接一下"。

### Step 3: 跑集成验证

能自动验证就自动验证。可以：

- 直接在某个会话里让它跑端到端测试 / 起服务冒烟测试；
- 或新开一个 `cc-integration` 会话专门做集成测试（前后端一起跑通一条主流程）。

```bash
ttmux send cc-web "起开发服务器，手动走一遍 注册→登录→访问受保护页 这条链路，报告每一步结果"
```

### Step 4: 对照需求规格验收

打开 `_spec.md` 的**验收标准**，逐条核对：

```markdown
## 验收核对（对照 _spec.md）
- [x] 注册/登录端点可跑通        ← cc-auth，pytest 7 case 全过
- [x] 前端登录页能拿到 token      ← cc-web，手动冒烟通过
- [x] RBAC 普通用户 403          ← cc-auth 测试覆盖
- [ ] OAuth Google 登录          ← ⚠️ 未实现，子任务漏了，需补
```

**有没达成的，就如实标出来**，不要假装完成。漏的可以再开会话补，或如实告诉用户。

### Step 5: 汇报用户

给用户一份收尾报告，包含：

- ✅ **达成了什么** —— 对照验收标准。
- 📂 **改了哪些东西** —— 各子任务的关键产出文件。
- ⚠️ **遗留 / 风险** —— 没做的、需要人工确认的、已知问题。
- 👉 **建议下一步** —— commit / 开 MR / 还需补什么。

> 记忆里有「不直接推 main，必须走 MR」的约定 —— 涉及提交时走 MR 流程，别直接 push main。

## 收尾

- 验收通过的子会话：记忆标记 done。是否 `ttmux kill` / `agent kill` 由用户决定，**不要擅自杀掉**用户可能还想看的会话，先问。
- `_spec.md` / `_tasks.md` 标记本次需求完结，保留作历史。

## 反模式

- ❌ 子会话各自说"我做完了"就直接报"全部完成" —— 没验对接、没对验收标准。
- ❌ 把验收标准里没达成的项悄悄略过。
- ❌ 擅自 commit / push / kill 会话 —— 这些动作问过用户或按既定约定来。
