# 拆任务 (decompose)

把 `_spec.md` 的需求拆成一组**可以分给独立子会话各自完成**的子任务。

> 好的拆分让多个会话能真正并行干活；坏的拆分让它们互相踩脚、改同一个文件冲突。

## 拆分原则

### 1. 按职责/模块切，不按工序切

- ✅ 好：`后端 API`、`前端页面`、`数据库迁移` —— 每块边界清晰，能独立完成。
- ❌ 差：`写代码`、`写测试`、`改 bug` —— 这是工序，没法并行，强行拆只会卡。

测试一般**跟着实现走**（同一个会话写完功能顺手写测试，见 test-push.md），
而不是单独拆一个"测试会话"。

### 2. 最小化跨任务的文件交叉

两个子任务如果会频繁改同一批文件 → 合成一个会话，或明确划分各自的文件边界。
**并行的前提是互不干扰。**

### 3. 显式标注依赖

有的任务必须等另一个先完成（如前端依赖后端的 API 契约）。画出依赖图：

```
[后端 API] ──┬──> [前端页面]
             └──> [集成测试]
[DB 迁移] ───────> [后端 API]
```

- **无依赖的** → 同时开（受并发上限约束，见 concurrency.md）。
- **有依赖的** → 先开上游，上游产出接口契约后再开下游（或下游先 mock）。

> **交给蜂群门控**：`ttmux swarm add <群> <成员> --depends-on a,b ...` 加的成员，
> 若依赖未完成会被自动**挂起为 pending**，不会立刻 spawn。你按任务图把全部成员一次加好即可，
> 不必手算开工顺序——上游被你 `ttmux swarm done <群> <上游>` 标记完成后，下游自动解锁。详见 patrol.md。

> **给每个成员定细分角色 + 职责**（组班子）：派活时带上 `--subrole` 与 `--duty`：
> ```bash
> ttmux swarm add <群> api --type agent \
>   --subrole backend --duty "实现推演引擎计算与数据接口" \
>   --depends-on arch "<起步任务>"
> ```
> `--subrole` 可选 `pm|architect|frontend|backend|fullstack|qa|designer|reviewer|devops|docs`（也可自定义字符串）；
> 它决定子会话拿到的**角色化 prompt**（产品偏需求/验收、测试偏用例/回归…），`--duty` 是该成员长期负责的范围。
> 拆任务时就想清楚"这支班子需要哪些角色"，别一律无差别 member。

> **把任务图落到看板**：拆完后，每个子任务在看板上建一张卡，作为全群可见的分工底账：
> ```bash
> t1=$(ttmux swarm task add <群> "认证后端" --desc "注册/登录/JWT")
> t2=$(ttmux swarm task add <群> "登录前端" --deps "$t1")
> ```
> 卡片是「要做的事」，成员会话是「干活的人」——开会话后用 `swarm task assign <群> <卡> <成员>` 把卡派给成员。
> **只有你(master)派活**；成员只认领、推进自己名下的卡。看板让全群随时看清谁负责什么、卡在哪。

### 4. 控制粒度

- 太粗（一个会话干整个需求）→ 失去并行价值，且单会话上下文过载。
- 太细（十几个微任务）→ 调度和集成成本爆炸，429 风险高。
- **经验值：2-5 个子任务**，每个是一个"一个工程师一两天能独立完成"的体量。

## 输出：任务图

把拆分结果写进蜂群记忆：

```
~/.local/share/ttmux/cc-swarm/_tasks.md
```

格式：

```markdown
# 任务拆分（对应 _spec.md）

## 子任务

### T1 · backend-auth  [会话: cc-auth]
- 职责：实现注册/登录/JWT 后端 API
- 依赖：无
- 产出：backend/routers/auth.py, backend/services/auth.py
- 文件边界：只动 backend/，提供 API 契约给 T2
- 验收：对应 _spec 验收标准的 第1、2条

### T2 · frontend-auth  [会话: cc-web]
- 职责：登录/注册页面 + 调 T1 的 API
- 依赖：T1（需要 API 契约）—— 可先按约定 mock 起步
- 产出：web/src/pages/auth/*
- 文件边界：只动 web/

### T3 · db-migration  [会话: cc-db]
- 职责：users 表迁移脚本
- 依赖：无（T1 依赖它，要先跑）
- 产出：backend/migrations/xxx.sql

## 依赖图
T3 ──> T1 ──> T2
       T1 ──> (集成测试，集成阶段做)

## 开工顺序
- 第一批（并行）：T3, T1 起步可并行（T1 先写不依赖表结构的部分）
- 第二批：T1 完成 API 契约后 → T2
```

## 会话命名约定

- 统一用 `cc-<简短职责>` 前缀，方便巡检时识别（如 `cc-auth`、`cc-web`、`cc-db`）。
- 名字要能一眼看出职责，别用 `cc-1`、`cc-2`。
- 记下来：哪个会话对应哪个子任务（写进 `_tasks.md` 和各自的记忆文件）。

## 拆完之后

- `plan` 模式：把任务图给用户看，等确认，**不开会话**。
- 完整流程：直接进入 spawn.md 开子会话。
- 拆分不是一锤定音 —— 巡检中发现某任务太大/有遗漏，可以再拆或追加会话，更新 `_tasks.md`。
