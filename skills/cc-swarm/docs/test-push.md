# 催测试 (test-push)

当 session 完成功能开发后，推动它写端到端测试。

## 触发条件

- session 状态为「已完成」（idle 在空 prompt，最近输出有完成总结）
- session 还没有写过测试（capture 中没有 test 相关输出）

## 指令原则

### 差异化 — 每个 session 的测试指令不同

**错误做法**（模板化）：
```
请写端到端测试。
```

**正确做法**（针对性）：
```
请为 OAuth 模块补充 pytest 测试。重点测3个场景：
1) 完整登录流程 - 注册→登录→拿token→访问受保护端点
2) Token刷新 - 旧token失效验证
3) RBAC权限 - 普通用户403
文件放 backend/tests/test_auth_flow.py
```

### 具体化 — 指定测试框架、场景、文件路径

从 capture 的上下文中提取：
- 它做了什么功能？→ 决定测什么
- 项目用什么测试框架？→ pytest / jest / vitest
- 已有的测试文件在哪？→ 保持一致的路径和命名
- 核心逻辑是什么？→ 重点测核心路径

### 间隔化 — 不要同时给所有 session 发测试指令

- 一次最多给 2 个发
- 两批之间间隔 30 秒以上
- 指令内容要有区分度（不同的措辞、不同的侧重点），避免429把所有的都堵了

## 指令模板（参考，实际使用时要根据上下文定制）

### 后端 API 测试
```
请为 [模块名] 写集成测试。用 pytest + httpx.AsyncClient，覆盖：
1) [核心流程1] — [具体验证点]
2) [核心流程2] — [具体验证点]
3) [边界case] — [具体验证点]
Mock 掉 [外部依赖]。文件放 backend/tests/test_[名字].py
```

### 前端组件测试
```
请为 [组件名] 写测试。用 vitest + @testing-library/react，覆盖：
1) [渲染场景] — [验证点]
2) [交互场景] — [验证点]
文件放 web/src/__tests__/[名字].test.tsx
```

### 数据处理测试
```
请为 [数据源/处理逻辑] 写单元测试，重点验证：
1) [正常数据流] — [验证点]
2) [降级/fallback] — [验证点]  
3) [边界条件] — [验证点]
用 unittest.mock patch 掉外部 API 调用。文件放 backend/tests/test_[名字].py
```

## 跟进

发完测试指令后，下一轮巡检关注：
- 它开始写了吗？还是 429 了？
- 写的过程中有审批需要批吗？
- 测试跑过了吗？有失败的需要修吗？
