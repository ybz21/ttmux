# 代码审查 (review)

对已完成的 session 做架构级 review，发现问题后发 challenge 指令。

## 触发条件

- session 已完成功能开发
- 还没有被 review 过（或修改后需要重新 review）

## Review 流程

### Step 1: 了解做了什么

从 capture 输出中提取：
- 创建/修改了哪些文件？
- 实现了什么功能？
- 整体架构是什么？

### Step 2: 读代码

用 Agent 工具启动 Explore 子 agent 来读代码：
- 读核心文件的实现
- 看接口设计、数据流、依赖关系
- 和已有代码对比风格一致性

### Step 3: 架构 Challenge

重点审查以下维度：

#### 职责划分
- 逻辑放对层了吗？（router → service → model）
- 有没有 service 做了 router 该做的事？反之？
- skill/工具层是否承担了太多业务逻辑？

#### 重复代码
- 有没有重新实现已有的函数？
- fallback/降级逻辑是否在多处重复？
- 相似的数据处理是否可以抽象？

#### 可配置性
- 硬编码的阈值/参数应该可配置吗？
- 不同场景需要不同配置怎么办？

#### 接口设计
- API 返回的数据结构是否明确（Pydantic model vs 自由 dict）？
- 前后端接口有没有契约（schema）？

#### 安全性
- token/密钥/敏感数据的处理是否安全？
- 有没有 timing-safe 比较？
- 输入校验是否充分？

#### 并发/状态
- 多实例部署时状态怎么共享？
- 有没有竞态条件？
- 缓存策略是否合理？

### Step 4: 发 Challenge 指令

发现问题后，给 session 发具体的修正指令：

**错误做法**：
```
代码有问题，请修改。
```

**正确做法**：
```
代码review了，有3个架构问题：

1. Fallback链在3处重复实现（service.py、intraday.py、sync.py）— 
   应抽一个 FallbackChain 类统一管理，调用方只要 router.get() 不关心降级。

2. 限流器60/min写死了 — 付费key应该自动用更高额度，从配置读取。

3. free tier能力判断分散在3个文件 — 应在 Source 类定义能力矩阵
   (supports_realtime, supports_intraday)，其他地方查矩阵。

请逐个确认并修正。
```

要点：
- **列出具体问题**，不说笼统的"有问题"
- **指出在哪里**，给出文件名和函数名
- **说明为什么有问题**，不只是"应该改"
- **给出方向**，但不给具体代码（让它自己想实现）

## 不需要 Challenge 的情况

- 代码简单直接，没有架构层面的问题
- 只是工具脚本、一次性迁移等
- session 已经在自我修正中
