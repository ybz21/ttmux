# 蜂群编排 (Swarm Orchestration) 设计文档

> 状态: **已定稿，进入实现**　作者: cc-swarm(神)　日期: 2026-06-15
>
> 命名: 对外统一叫 **蜂群 / swarm**（不再叫「任务组 / 群组」）。

---

## 1. 背景与动机

今天 ttmux 有两套割裂的东西：

1. **任务编排**（ttmux 核心）—— `.group` 文件，把一批并行 tmux 会话归为一组。
   成员可以是普通 shell 命令，也可以是 Claude Agent。但 `.group` 只是**会话名清单**，
   没有目标、没有类型、没有依赖、没有「谁在管它」。
2. **cc-swarm / 神**（监护 skill）—— 总指挥，巡检所有 `cc-*` 会话做审批/催测试/review/集成。
   但监护范围是**全局的**，没法说「只盯这一摊活」。

用户诉求：

> **任务编排 ≈ 一个「蜂群」概念。可以建一个蜂群，然后 cc 操作这个蜂群。**

把两者合一：**蜂群 = 一摊有共同目标的活（一批会话 + 目标 + 依赖 + 状态）；
cc 操作蜂群 = 把 cc-swarm 的监护收敛到「某一个蜂群」。**

一句话：**蜂群就是一群 agent，cc 是这个蜂群的指挥。**

## 2. 已定决策

| # | 决策 | 选择 |
|---|------|------|
| 1 | cc 接管蜂群的方式 | **`ttmux swarm adopt` 自动拉起 cc** —— 一条命令写 supervisor + 开/复用指挥会话并带蜂群作用域跑 `/cc-swarm` |
| 2 | 本轮范围 | **一路做到阶段 3**：数据模型 → 两层界面 → cc 作用域巡检 |
| 3 | 命名 | **蜂群 / swarm**（中文「蜂群」、命令 `ttmux swarm`、内部 swarm） |

默认（未单独问、按合理取值）：
- 蜂群元数据**复用** cc-swarm 现有记忆目录 `~/.local/share/ttmux/cc-swarm/`，蜂群清单存 `swarms/`。
- `depends_on` 字段**纳入模型但可为空**；本轮先支持无依赖并行，依赖解锁作为阶段 3 末尾增量。

## 3. 核心概念

| 概念 | 定义 |
|------|------|
| **蜂群 (Swarm)** | 有名字、有目标的工作容器。内含若干**成员**，可被 cc 接管监护。 |
| **成员 (Member)** | 一个工作单元 = 一个 detached tmux 会话。两类：`task`（shell）/ `agent`（Claude）。 |
| **目标 (Goal)** | 蜂群要达成的事，自由文字 + 可选验收标准。cc 集成时对照它。 |
| **指挥 (Supervisor)** | 接管该蜂群的 cc-swarm 会话（如 `god`）。一个蜂群同时最多一个指挥。 |
| **蜂群状态** | `planning` → `running` → `integrating` → `done` / `archived`。 |

两类成员**可共存于同一蜂群**（如 3 个 agent 写代码 + 1 个 task 跑 watch 测试）。

---

## 4. 交互模型：两层

今天菜单是扁平的；新模型**两层**，贴合「蜂群」概念：

```
任务编排  (第一层 — 蜂群管理)
  • 列出蜂群          ← 每行：名字 / 目标 / 成员数 / 状态 / 指挥
  • 新建蜂群          ← 起名 + 写目标(可选)
  • 进入蜂群 ▸        ← 钻进某蜂群，进第二层
  • cc 接管蜂群       ← adopt：交给 cc-swarm 监护（核心新能力）
  • 归档/清理蜂群

  蜂群详情  (第二层 — 成员管理)
    • 加成员: 普通任务 / Claude Agent
    • 成员状态（自动识别类型，agent 显示任务摘要）
    • 追加指令（对某成员）
    • 等待并收集
    • 清理成员 / 整群
```

> 第一层管「蜂群」，第二层管「蜂群里的成员」。
> 「建一个蜂群、再操作它」就是自然的钻取路径。

## 5. 数据模型

### 现状
```
~/.local/share/ttmux/groups/<group>.group   # 纯文本，每行一个会话名
~/.local/share/ttmux/agents/<session>/       # agent 元数据(task/workdir/started)
```

### 提案：蜂群清单 (manifest)，向后兼容旧 `.group`
```
~/.local/share/ttmux/swarms/<swarm>/
├── swarm.json            # 蜂群级元数据
└── members/<name>.json   # 每个成员的元数据
```

`swarm.json`：
```json
{
  "name": "user-system",
  "goal": "实现用户系统：注册登录、OAuth、RBAC",
  "status": "running",
  "supervisor": "god",
  "created": "2026-06-15 18:00:00"
}
```

`members/<name>.json`：
```json
{
  "name": "api",
  "type": "agent",
  "session": "user-system-api",
  "task": "重构用户认证模块",
  "workdir": "/home/ai/proj",
  "depends_on": [],
  "status": "running"
}
```

**新增关键字段**：`type`（省去靠探测 `agents/` 目录的 hack）、`goal`、`supervisor`、
`depends_on`（依赖图）、`status`。

### 向后兼容
- 旧扁平 `<group>.group` 仍可读：当作「无目标、成员全 task、无依赖、无指挥」的蜂群。
- 首次访问旧 `.group` 时**自动升级**为新目录结构（保留双读，不破坏老数据）。

## 6. cc 如何操作蜂群（核心）

把 cc-swarm 生命周期映射到蜂群，并**收敛范围到单个蜂群**：

| cc-swarm 阶段 | 蜂群模型里的动作 |
|---------------|------------------|
| 接需求/建需求 | 写 `swarm.json.goal`(+验收标准) |
| 拆任务 | 生成 `members/*.json`(含 type/depends_on)，画依赖图 |
| 开子会话 | 按成员清单 spawn（task→`_do_spawn`，agent→`_agent_spawn`） |
| **巡检** | **只 capture 本蜂群成员**（不再全局扫 `cc-*`）；审批/催测试/解锁下游 |
| 集成验收 | `collect` 全部成员输出，对照 `goal` 验收，置 `status=done` |

**`ttmux swarm adopt <swarm>`（决策 1）**做三件事：
1. 在 `swarm.json` 写 `supervisor=<cc会话>`、`status=running`；
2. 开（或复用）一个指挥用的 cc 会话；
3. 在该会话注入 `/cc-swarm --swarm <swarm>`，进入**蜂群作用域巡检循环**：
   - 探测成员 = 读 `members/*.json` 的 session（而非 `ttmux ls` 全表过滤）；
   - 依赖解锁 = 成员完成后按 `depends_on` 启动下游；
   - 自我排除 = 跳过 supervisor 自己；
   - 多蜂群隔离 = 不同 cc 会话各管各的蜂群，互不干扰。

> cc 监护从「全局猜哪些是我的活」→「明确我负责这个蜂群」，可并行多个独立蜂群、各有指挥。

## 7. 命令 / 接口（提案）

保留现有 `spawn` / `agent` / `group` / `collect` 不变（向后兼容），**新增** `swarm` 子命令：

```
ttmux swarm new <名> [--goal "..."]              # 建蜂群
ttmux swarm ls                                   # 列蜂群(目标/状态/指挥)
ttmux swarm add <群> <名> --type task  <命令>
ttmux swarm add <群> <名> --type agent <任务> [--dir/--perm/--model]
ttmux swarm adopt <群> [--by <cc会话>]           # cc 接管(自动拉起 cc + 作用域巡检)
ttmux swarm status <群>                          # 成员状态(自动识别类型)
ttmux swarm collect <群>                         # 收集成员输出
ttmux swarm done|archive <群>
```

`spawn ...` / `agent spawn ...` 视为 `swarm new` + `swarm add` 的**快捷方式**（语义不变）。
cc-swarm skill 的 `SKILL.md` 新增 `--swarm <名>` 作用域参数，patrol.md 改为读 manifest。

## 8. 分阶段实施计划（本轮做到阶段 3）

> 实现落点：源码已拆到 `ttmux-cli/lib/`，蜂群逻辑集中在 `lib/swarm.sh`，
> `build.sh` 合并成根 `ttmux`。实际存储用**纯文本元数据文件**（与既有 `_task_*meta` 风格一致），
> 不用 JSON 文件（避免 jq 依赖）；`--json` 输出仍走既有 python 路径。

- **阶段 0 ✅**：交互菜单合并「任务编排 + 多 Agent」→ 单一「任务编排」段，
  状态/收集/清理统一走 `_task_*` 元数据（仓库 v0.4.0 已具备）。
- **阶段 1 ✅**：蜂群元数据模型 `swarms/<名>/{goal,status,supervisor,created}.txt` + `deps/<成员>.txt`；
  命令 `swarm new/add/ls/status/collect/done/archive/rm`。成员复用 `_spawn_one`/`_status`/`_collect`/`_kill_group`。
- **阶段 2 ✅**：两层交互界面——第一层 `_interactive_swarm`（列表/新建/进入/接管/归档），
  第二层 `_interactive_swarm_detail`（加成员/追加指令/收集/清理/接管）。主菜单 5 号入口进入。
- **阶段 3 ✅**：`swarm adopt`（写 supervisor + 拉起或复用交互式 cc 会话，注入 `/cc-swarm --swarm <名>`）；
  cc-swarm skill 新增 `--swarm <名>` 作用域，`docs/patrol.md` 增「蜂群作用域」（读 `swarm status` 而非全局扫）。
  - **`depends_on` 现状**：作为元数据记录并在 `swarm status` 展示，由 cc 指挥**按依赖图调度**（AI 驱动）。
    成员目前在 `swarm add` 时**即时启动**，尚未做「依赖未满足则延迟 spawn」的自动门控——
    这是明确的后续增量（需要 pending 成员存储 + `swarm activate`）。

## 9. 残留小问题（实现中按默认推进，可随时纠偏）

1. 指挥会话用什么名字？默认 `cc-<swarm>` 或复用当前 `god`，`adopt --by` 可指定。
2. `swarm archive` 是否保留日志？默认保留 manifest + 日志，仅置 `status=archived`、kill 会话。
