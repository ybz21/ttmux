# 国际化规范

状态：新工作强制执行。已有硬编码文案在触碰相关文件时顺手迁移；专门做国际化改造时再集中清理。

Roam 是国际化产品界面。开发讨论和源码注释可以继续以中文为主，但产品 UI 必须至少支持简体中文和英文。

## 1. 适用范围

本规范覆盖所有用户可见文案：

- `frontend/src/**`
- `frontend/index.html` 与 `frontend/public/**`
- `backend/server/fallback.html`
- `backend/home/index.html`
- 会被前端直接展示的后端 API 消息
- 产品 UI 内展示的 README 或文档链接文案

以下内容不要求翻译：

- 用户 shell 里的终端输出
- Claude、Codex 或其他 Agent 的对话正文
- 文件内容、路径、命令、环境变量名、日志、堆栈、协议字段
- 内部代码注释和仅开发者阅读的文档
- 品牌或专有名词，例如 `Roam`、`ttmux`、`Claude Code`、`Codex`、`tmux`、`Chrome`

## 2. 支持语言

必须支持：

- `zh-CN`：简体中文
- `en-US`：英文

当前产品文案以中文为主，`zh-CN` 作为源语言。任何新增 key 必须在同一个变更里同时提供 `zh-CN` 和 `en-US`。

语言选择顺序：

1. 用户显式设置，持久化到 `localStorage`
2. 浏览器语言，且命中支持语言
3. 回退到 `zh-CN`

## 3. 前端架构标准

前端必须有一层统一 i18n，不允许各组件各自维护翻译表。实现或扩展位置固定为 `frontend/src/i18n/`。

推荐目录：

```text
frontend/src/i18n/
├── index.tsx              Provider、hook、语言检测、持久化
├── locales/
│   ├── zh-CN.ts
│   └── en-US.ts
└── keys.ts                可选：共享 key/type helper
```

React 根节点必须先挂 i18n provider，再渲染业务组件。Ant Design 的 `ConfigProvider` locale 必须跟随同一份语言状态，不能固定写死 `zh_CN`。

新增业务模块必须从第一行 UI 代码开始接入 `useI18n()`，不能先写中文再“以后统一替换”。模块目录里不允许私有翻译表；所有产品文案必须进入全局 locale 文件，确保语言切换、审计脚本和缺失 key 检查覆盖得到。

要求暴露的基础 API：

```tsx
const { t, locale, setLocale } = useI18n()

t('nav.sessions')
t('session.status.running')
t('file.deleteConfirm', { name })
```

优先使用项目内轻量实现。若后续引入 `react-i18next` 等库，也必须遵守本规范里的 key、插值和覆盖率要求。

## 4. Key 命名

key 必须是稳定的语义标识，不能拿中文或英文原句当 key。

正确：

```ts
'nav.sessions'
'session.action.new'
'session.status.running'
'file.preview.unsupported'
```

错误：

```ts
'会话'
'New session'
'button1'
'text.delete.file.modal.title'
```

按产品域分 namespace：

- `app.*`
- `nav.*`
- `auth.*`
- `overview.*`
- `session.*`
- `terminal.*`
- `chat.*`
- `swarm.*`
- `file.*`
- `browser.*`
- `env.*`
- `settings.*`
- `common.*`

只复用真正通用的词和动作。不要因为当前中英文刚好一样，就把不同业务语境的文案硬复用到一个 key。

## 5. UI 文案写法

React JSX、前端常量、toast、modal、placeholder、tooltip、表头、空状态、校验提示里不得新增硬编码用户可见中英文。

错误：

```tsx
<Button>刷新</Button>
message.success('已保存')
<Input placeholder="输入会话名" />
```

正确：

```tsx
<Button>{t('common.refresh')}</Button>
message.success(t('common.saved'))
<Input placeholder={t('session.namePlaceholder')} />
```

带变量的文案必须把完整句子放进语言包：

```tsx
t('session.failedWithCode', { code })
```

不要拼接本地化语法：

```tsx
// 错误
t('session.failed') + ' ' + code
```

## 6. 日期、数字和列表

用户可见的日期、数字、百分比、列表连接词必须使用当前语言：

- `Intl.DateTimeFormat`
- `Intl.NumberFormat`
- `Intl.ListFormat`
- `Intl.RelativeTimeFormat`

不要在用户界面里硬编码 `YYYY-MM-DD HH:mm` 一类格式，除非它是机器时间戳、日志时间、文件名或协议字段。

## 7. Ant Design

Ant Design locale 必须跟随应用语言：

- `zh-CN` -> `antd/locale/zh_CN`
- `en-US` -> `antd/locale/en_US`

业务组件不得直接 import 固定 Ant Design locale。映射逻辑统一放在 provider/theme 层。

## 8. 后端和静态 HTML

后端 API 错误优先返回稳定机器码和可选默认消息。前端已知错误码由前端翻译。

直接返回给浏览器的静态 HTML 必须满足其中之一：

- 内置最小语言表，并根据 `navigator.language` 选择语言
- 只保留泛化的运行/故障兜底文案，不写大量产品操作文案

如果某个静态兜底页面暂不国际化，必须在任务总结或 PR 里写明这是已知例外。

## 9. 迁移策略

新功能：

- UI 代码同变更添加翻译 key
- `zh-CN` 与 `en-US` 同时补齐
- 涉及日期选择器、分页、上传、空状态、校验提示时同步检查 Ant Design locale

改已有文件：

- 修改到的区域如果有硬编码 UI 文案，应顺手迁移
- 组件文案很集中时，优先迁完整组件
- 不要在非国际化任务里制造大面积纯文案 churn

建议落地顺序：

1. 建 `frontend/src/i18n/`、语言包和 `useI18n()`
2. 改造 `ThemeProvider` / 根 provider，让 Ant Design locale 跟随应用语言
3. 先迁导航、登录、全局 toast、通用按钮和状态标签
4. 再按页面迁移：会话、终端、文件、浏览器、蜂群、系统配置
5. 最后处理静态 fallback HTML 和后端可见错误码

## 10. 验收清单

完成前端或用户可见后端改动前必须检查：

- 没有新增硬编码用户可见中文或英文
- 新增 key 在 `zh-CN` 和 `en-US` 都存在
- 插值使用命名变量，不用位置参数
- 日期、数字、列表使用当前语言
- Ant Design locale 跟随应用语言
- 改动过的页面在 `zh-CN` 和 `en-US` 下都做过基本检查；有布局风险时要截图验收

强制审计命令：

```bash
cd frontend
npm run i18n:check
```

`npm run build` 会先执行 `npm run i18n:check`，任何新增模块如果出现以下问题都应失败：

- `zh-CN` 和 `en-US` key 不一致
- JSX 属性、弹窗、toast、空状态、placeholder 中出现硬编码用户可见文案
- 可疑中文 UI 文案没有走 `t(...)`

GitHub PR/MR 也必须通过 `Frontend i18n audit` workflow。该检查会在涉及 `frontend/**` 或本规范变更时执行 `npm ci`、`npm run i18n:check` 和 `npm run build`。仓库维护者应把这个 workflow 配成分支保护的 required status check，避免未国际化 UI 合入主干。

辅助人工扫描命令：

```bash
rg -n "[\u4e00-\u9fff]|>[A-Za-z][^<{]*<|placeholder=\"|title=\"" frontend/src backend/server/fallback.html backend/home/index.html
```
