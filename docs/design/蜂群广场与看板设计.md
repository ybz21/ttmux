## 1. 背景与动机

现在一个蜂群里，成员之间、成员与 master 之间的协作**全靠抓终端**：
master 用 `ttmux capture <成员>` 读屏、用 `ttmux send` 发指令。这有三个硬伤：

1. **无共享上下文**：成员 A 的产出（"API 契约定了：`POST /auth/login → {token}`"）成员 B 看不到，
   除非 master 人肉转述。没有一块所有人都能读写的"黑板"。
2. **巡检信号脆弱**：靠正则/AI 读屏判断"它在干嘛"，终端一 `/clear` 或刷屏就丢上下文。
3. **任务图不可见**：谁负责什么、做到哪一步、卡在哪，散落在 `deps/` + `pending/` + 各终端里，
   没有一个**统一的、可派活、可流转**的任务视图。

引入两个**每个蜂群各一份、实例化落盘**的协作子系统：

| 子系统 | 类比 | 解决 |
|--------|------|------|
| **广场 (Plaza)** | 团队频道 / 留言板 | 所有人发言、所有人可读的**异步消息流**。结构化的协作信号，替代抓终端。 |
| **看板 (Board)** | 看板 (Kanban) | 把"要做的活"做成**卡片**，可派给成员、可在列间流转。统一的任务图。 |

一句话：**广场是蜂群的「嘴」(沟通)，看板是蜂群的「手」(分工)。** master 用看板派活、用广场读心。

### 核心定位（一句话钉死）

> **多个智能体(成员)通过「广场 + 看板」协作；但分工不是去中心化的——有且只有一个「主控智能体(master)」负责把大目标拆成卡片、派给成员。**

- **协作是多边的**：任何成员/master/human 都能在广场发言、都能读看板，信息对所有人可见。
- **分配是单中心的**：**只有 master 往看板派活(`task add`/`assign`)**。成员**不自行给别人派活**，
  只认领自己名下的卡、推进它、把进展播报到广场。master 据广场+看板做下一步分配与调度。
- 三者分明：**human** 给目标/拍板；**master** 拆解+分配+调度+验收；**成员** 执行+协作沟通。

## 3. 核心概念

| 概念 | 定义 |
|------|------|
| **广场 (Plaza)** | 蜂群级**追加型(append-only)消息流**。一条消息 = 一次发言。所有成员/master/human 都能发、都能读。 |
| **言论 (Post)** | 广场里的一条消息：`{id, ts, author, kind, re?, text}`。`kind` 用于过滤(见下)。 |
| **看板 (Board)** | 蜂群级**任务卡片集合**，按列(column)组织。 |
| **卡片 (Card / Task)** | 一个工作单元：`{id, title, desc?, assignee?, col, deps?, created, updated}`。**卡片是"要做的事"，成员是"干活的会话"**，多对一。 |
| **列 (Column)** | 卡片所处阶段：`backlog`(待办) → `assigned`(已派) → `doing`(进行) → `review`(待审) → `done`(完成)。另有 `blocked`(受阻)。 |

> **看板卡片 vs 成员**：刻意分开。一个 `agent` 成员(会话)可在生命周期里先后认领多张卡片；
> 一张卡片也可重派给别的成员。卡片承载"任务语义+流转"，成员承载"算力+会话"。
> master 负责把卡片 ↔ 成员对应起来（见 §7）。

## 4. 数据模型（落盘）

在既有蜂群目录下新增 `plaza/` 与 `board/`：

```
~/.local/share/ttmux/swarms/<群>/
├── goal.txt  status.txt  supervisor.txt  created.txt     # (既有)
├── deps/<成员>.txt   pending/<成员>/...   done/<成员>       # (既有, 依赖门控)
│
├── plaza/                          # ── 广场 ──
│   ├── feed.log                    # 追加型消息流, 每行一条 (TSV)
│   └── seq                         # 自增消息号计数器
│
└── board/                          # ── 看板 ──
    ├── seq                         # 自增卡片号计数器
    └── <卡id>.card                 # 每张卡一个文件 (key=value)
```

### 4.1 广场 `plaza/feed.log`（TSV，追加型）

一行一条，制表符分隔，**消息体单行**（换行转义为 `\n`，渲染时还原）：

```
<id>\t<ts>\t<author>\t<kind>\t<re>\t<text>
```

示例：
```
1	2026-06-16 14:02:11	master	broadcast	-	开工。看板已派活, 各成员认领自己的卡。
2	2026-06-16 14:08:33	api	done	-	认证后端完成, 契约: POST /auth/login -> {token}
3	2026-06-16 14:09:01	web	ask	2	token 放 header 还是 body?
4	2026-06-16 14:10:20	master	decide	3	放 Authorization: Bearer。
```

- **追加型**：只 append，不改历史 → 天然并发安全（多个成员同时 `say` 用 `>>` 原子追加），
  也是 `watch`(tail -f) 与 web SSE 的理想后端。
- `id`：取自 `plaza/seq` 自增。
- `kind`（可过滤）：`note`(默认/随手记) `ask`(提问) `block`(报阻塞) `decide`(决策) `done`(完成播报) `broadcast`(全员广播)。
- `re`：可选，引用某条消息 id（形成问答链），无则 `-`。

### 4.2 看板 `board/<卡id>.card`（key=value）

每张卡一个文件，与既有 `_swarm_meta_*` 同款 key=value 风格，便于单字段改写：

```
title=前端登录页
desc=登录/注册页 + 调 api 的契约
assignee=web
col=doing
deps=t1
created=2026-06-16 14:01:00
updated=2026-06-16 14:09:30
```

- `卡id`：`t<N>`，N 取自 `board/seq` 自增（`t1`, `t2`…）。
- `assignee`：成员名(`<群>-<成员>` 的 `<成员>` 部分)，可空。
- `col`：列名，默认入 `backlog`。
- `deps`：逗号分隔的**卡片 id**（卡片间依赖，区别于成员 `deps/`）。仅供看板展示/排序，不自动门控成员。

### 4.3 与既有依赖门控的关系（重要）

**两套依赖，互不打架**：
- 成员 `deps/<成员>.txt` = **会话级**依赖 → 驱动 pending/activate **自动门控**（我已实现）。
- 卡片 `deps=` = **任务级**依赖 → 仅看板内排序/可视化，**不自动 spawn**。

桥接由 master 显式完成，不做隐式魔法（见 §6 松耦合原则）。

## 5. CLI 命令

### 5.1 广场

```
ttmux swarm say   <群> [--as <成员>] [--kind <类型>] [--re <id>] <消息>
        # 发一条言论。author 默认自动署名(见 §8), --as 覆盖。
ttmux swarm feed  <群> [-n <N>] [--from <成员>] [--kind <类型>] [--since <id>] [--json]
        # 读消息流。默认末 N=30 条; --since <id> 取该 id 之后(增量轮询用)。
ttmux swarm watch <群>
        # 实时跟随(tail -f feed.log + 渲染), 给 master 盯场 / web SSE。
```

示例：
```bash
# 成员在自己的会话里播报完成 (自动署名为 api)
ttmux swarm say myapp --kind done "认证后端完成, 契约 POST /auth/login -> {token}"
# master 读全场, 只看提问和阻塞
ttmux swarm feed myapp --kind ask
ttmux swarm feed myapp --kind block
# 增量轮询(web/巡检): 取上次看到的 id 之后的新消息
ttmux swarm feed myapp --since 12 --json
```

### 5.2 看板

```
ttmux swarm board <群> [--json]
        # 渲染看板: 按列分组展示所有卡片 (列 × 卡片网格)。

ttmux swarm task add    <群> "<标题>" [--desc "..."] [--assignee <成员>] [--deps t1,t2] [--col <列>]
        # 新建卡片, 打印新卡 id。默认入 backlog。
ttmux swarm task ls     <群> [--col <列>] [--assignee <成员>] [--json]
        # 平铺列出卡片 (可按列/责任人过滤)。
ttmux swarm task show   <群> <卡id>                  # 卡片详情
ttmux swarm task assign <群> <卡id> <成员>           # 派活: 设 assignee + 自动移到 assigned
ttmux swarm task move   <群> <卡id> <列>             # 流转: backlog/assigned/doing/review/done/blocked
ttmux swarm task done   <群> <卡id>                  # = move ... done 的快捷
ttmux swarm task rm     <群> <卡id>                  # 删卡
```

示例：
```bash
# master 拆活上板
t1=$(ttmux swarm task add myapp "认证后端" --desc "注册/登录/JWT")
t2=$(ttmux swarm task add myapp "登录前端" --deps "$t1")
# 派给成员 (成员会话由 swarm add 创建)
ttmux swarm task assign myapp "$t1" api
ttmux swarm task assign myapp "$t2" web
# 成员认领后自己推进
ttmux swarm task move myapp "$t1" doing
# 看全局
ttmux swarm board myapp
```

> **命名消歧**：`swarm done <群> [成员]`(标记成员/整群完成, 既有) 与 `swarm task done <群> <卡id>`(标记卡片完成)
> 在不同命名空间(`done` vs `task done`)，不冲突；`swarm add`(加成员) 与 `swarm task add`(加卡片) 同理。

## 6. 设计原则：松耦合 + 显式桥接

广场、看板、成员门控是**三个正交机制**，各自独立可用、互不隐式联动。串联由 master(AI) **显式**做：

- 加卡 ≠ 开成员。`swarm task add` 只在看板建卡；要干活得 master 另行 `swarm add` 开成员会话。
- 标卡完成 ≠ 解锁成员下游。`swarm task done` 只动看板；成员依赖解锁仍走 `swarm done <群> <成员>`。
- 发言 ≠ 任何状态变更。`swarm say` 纯沟通。

**可选便捷桥接（默认关闭，P2 增量，避免魔法）**：
- `swarm done <群> <成员> --with-cards` —— 标记成员完成时，顺手把该成员 `doing`/`review` 的卡移到 `done`。
- `swarm task assign <群> <卡id> <成员> --spawn` —— 派卡时若成员会话不存在, 顺手 `swarm add` 开一个并把卡片标题/描述作为任务。
- 卡片移入 `done`/`blocked` 时自动往广场 `say` 一条 `kind=done`/`block`（可由全局开关 `TTMUX_BOARD_NOTIFY=1` 打开）。

> 原则：**核心 CLI 保持哑而正交，编排智能放在 master(cc-swarm skill)**。这与蜂群"所有决策由 AI 做"一致。

## 7. 角色协作流（master / 成员 / human 怎么用）

```
human ──给大目标──▶ master(cc-swarm)
                      │① 拆活上板:  swarm task add ... (每个子任务一张卡, 卡间 deps)
                      │② 派活+开人: swarm task assign + swarm add (成员会话, 含成员级 deps 门控)
                      │             并 swarm say --kind broadcast "开工, 各认领自己的卡"
                      ▼
   成员 api / web ... 各自:
       - swarm task ls <群> --assignee <我>      # 我有哪些卡
       - swarm task move <群> <卡> doing          # 认领推进
       - 干活... 有契约/产出 ──▶ swarm say --kind done/note "..."   # 播报到广场, 别人可读
       - 卡住 ──▶ swarm say --kind block "缺 X"   # master 看到 block 介入
       - 做完 ──▶ swarm task move <群> <卡> review
                      ▲
                      │③ 巡检(patrol) = 读广场 + 看板, 不再纯抓终端:
   master 循环:        │     swarm feed <群> --since <上次id>   # 新消息(完成/提问/阻塞)
                      │     swarm board <群>                    # 任务全貌
                      │   决策: 批准/答疑/review/解锁:
                      │     - 成员真完成 ──▶ swarm done <群> <成员>   # 解锁下游 pending 成员
                      │     - 卡过审    ──▶ swarm task done <群> <卡>
                      ▼
                   全部卡 done + 对照 goal 验收 ──▶ swarm done <群> ──▶ 汇报 human
```

**关键升级**：巡检信号从"抓终端猜状态" → "**读广场(结构化播报) + 看板(任务状态)**"。
终端仍可作兜底(深看某成员在干嘛)，但日常协作走广场/看板，鲁棒得多。

## 8. 自动署名（author 推断）

`swarm say` 不传 `--as` 时，按当前 tmux 会话名推断作者，零配置归属：

| 当前会话名 | 推断 author |
|-----------|------------|
| `<群>-<成员>`（成员会话）| `<成员>` |
| `cc-<群>` / 蜂群 supervisor | `master` |
| 其它（裸 shell / web 后端代发）| `human` |

实现：`tmux display-message -p '#{session_name}'`（在 pane 内才有值）。
`--as <名>` 始终覆盖推断（web 后端代某成员发言时用）。

## 9. Web 映射（ttmux-web）

两个子系统天然对应两个面板，复用既有基建：

| 子系统 | 前端 | 后端(转发 ttmux) | 实时 |
|--------|------|------------------|------|
| **广场** | 聊天/动态流面板(气泡含 author/kind 色标) | `GET /api/swarms/:n/feed?since=` ; `POST /api/swarms/:n/say` | `swarm watch` → SSE(复用 `/api/logs` 同款 hub) |
| **看板** | Kanban 列+卡(可拖拽流转) | `GET /api/swarms/:n/board` ; `POST .../task` ; `PATCH .../task/:id`(assign/move) | 轮询 `board --json` 或 SSE |

human 在 web 广场直接发言(`--as human`)即可参与协作；拖卡 = `task move`。
这与上一轮《建群+master 仪表盘》方案里的"仪表盘"自然合并：仪表盘 = master 终端 + 成员网格 + **广场** + **看板** + 依赖图。

## 11. 残留小问题（实现时按默认推进，可纠偏）

1. **广场消息体含换行/制表符**：默认转义 `\n`/`\t` 存、渲染时还原；超长(>2KB)截断并提示。
2. **feed.log 增长**：默认不滚动；`archive` 时随蜂群一起归档。需要再加 `swarm say --rotate` 或按行数滚动(后续)。
3. **卡片列集合是否可定制**：默认固定 6 列(`backlog/assigned/doing/review/done/blocked`)，不开放自定义(简单优先)。
4. **并发写 feed.log**：依赖 `>>` 的原子追加(单行 <4KB 在 Linux 上原子)，消息体强制单行即可，无需锁。
5. **卡片 deps 是否参与看板拓扑排序**：默认仅展示"卡A依赖卡B"标注，不重排；可视化拓扑排序留 P3。
