# 蜂群广场监听循环设计

> 状态：**设计稿，待实现**　作者：Codex　日期：2026-06-21
>
> 前置：建立在 [蜂群编排设计.md](蜂群编排设计.md)、
> [蜂群广场与看板设计.md](蜂群广场与看板设计.md)、
> [蜂群成员启动与提示词模板设计.md](蜂群成员启动与提示词模板设计.md) 之上。

## 1. 问题

当前 swarm 里，成员完成后会把结果推到广场 / 看板；human 也可以在看板或广场里继续发消息。
但 master 的巡检主要由自身 loop 驱动，若它已经判断“完成”或进入低频等待，human 在看板里追加的新指令
可能不会被及时读取、判断和分发。

要补的不是“消息通知 UI”，而是一个**每个 agent 常驻的 Plaza Listener Loop**：

- **master 必须持续监听广场中的 human 指令**，即使蜂群看起来已经完成，也要重新判断是否需要 reopen / 派活 / 答复。
- **worker 也要监听广场中与自己相关的指令**，尤其是 master 的派活、裁决、返工要求和 human 的点名补充。
- 监听必须有去重状态，避免同一条消息被反复执行。
- 监听应该优先走结构化广场消息，终端 capture 只做兜底。

一句话：**广场是蜂群的控制面；所有 claude/codex agent 都要有读控制面的 loop，master 对 human 消息负最终判断责任。**

## 2. 目标与非目标

### 目标

1. master 长驻监听本蜂群广场，优先处理 `author=human` 的新消息。
2. worker 长驻监听广场，处理点名自己、自己卡片相关、或 master 广播的消息。
3. 引入统一的消费游标，记录每个 listener 已处理到的 post id。
4. human 指令进入后，master 能基于当前看板 / 状态做判断：答复、派活、返工、验收、重开已完成蜂群。
5. 兼容 claude / codex 两种 agent，不依赖模型厂商能力。

### 非目标

- 不把所有决策写成脚本规则。脚本只负责拉消息、存游标、提供 prompt；是否执行由 agent 理解后决定。
- 不要求 WebSocket 真推送。M1 用轮询 `swarm feed --since` 即可，后续可升级为事件流。
- 不让 worker 自行给其他 worker 派活。worker 可读消息、更新自己的卡、汇报阻塞；全局调度仍归 master。

## 3. 核心机制

### 3.1 角色职责

| 角色 | 监听范围 | 必须处理 | 可以忽略 |
|------|----------|----------|----------|
| master | 全量新消息，特别是 human | human 指令、worker ask/block/done、review 列卡片、已完成后新需求 | 与决策无关的普通 note |
| worker | 与自己相关的新消息 | `@自己`、分配给自己的卡、master decide/broadcast、返工要求 | 其他 worker 的普通进度 |
| human | 不跑 loop | 通过广场 / 看板发需求、追问、拍板 | - |

### 3.2 消息分类

现有 `posts` 表已有 `author/kind/re/text`。不强制新增字段，M1 用约定增强：

- `author=human`：最高优先级，master 每轮必须读。
- `kind=ask`：需要答复。
- `kind=block`：需要解阻塞。
- `kind=done`：触发 review / 验收。
- `kind=decide`：master 裁决，worker 需要跟进。
- `kind=broadcast`：全员广播。
- 文本中的 `@成员名`：点名路由。
- 文本中的 `@master`：定向给 master，通常来自 human 追加需求、追问、叫停或验收反馈。
- 文本中的 `@all`：全员可见且都应判断是否相关，但仍由 master 负责全局调度。
- 文本中的 `#t1` / `卡 t1`：卡片路由。

P2 可给 `posts` 增加 `target` / `priority` / `handled_by` 字段，但 M1 不依赖 schema 变更。

### 3.3 @xx 提及机制

`@xx` 是广场的**定向提及能力**，解决“消息发到广场了，但应该谁处理”的问题。
它不是新的 `kind`，而是和 `kind` 正交：

```bash
ttmux swarm say <群> --as human --kind ask "@master 这个需求完成后还要补移动端验收"
ttmux swarm say <群> --as master --kind decide "@web #t4 这张卡返工，补空状态"
ttmux swarm say <群> --as api --kind block "@master #t2 缺少登录接口字段定义"
ttmux swarm say <群> --as master --kind broadcast "@all 今天先不要改数据库 schema"
```

#### 提及语法

| 写法 | 含义 |
|------|------|
| `@master` | 指挥 / supervisor 必须处理 |
| `@human` | 需要 human 拍板或补充信息 |
| `@all` | 所有 agent 都要读，但不能因此越权派活 |
| `@<成员>` | 点名某个成员，例如 `@api`、`@web` |
| `@<角色>` | P2 支持角色别名，例如 `@frontend` 路由到一组成员 |

#### 路由规则

M1 用文本解析即可，不要求 schema 变更：

1. master 监听全量消息，`@master` / `author=human` / `@all` 优先级最高。
2. worker 只强制处理 `@自己`、`@all`、自己卡片 `#tN`、master 对自己的回复链。
3. human 发普通消息但没有 `@master` 时，master 仍要读；Web 端可以自动补 `@master`。
4. `@成员 + #卡id` 同时出现时，worker 以卡片上下文处理。
5. 找不到成员时，master 在广场回 `kind=ask|decide` 说明无法路由，而不是静默忽略。

#### CLI 支撑

保留现有文本写法即可，同时建议新增糖：

```bash
ttmux swarm say <群> --to master --kind ask "这个需求还要补移动端"
```

等价于：

```bash
ttmux swarm say <群> --kind ask "@master 这个需求还要补移动端"
```

M2 的 `swarm listen` 可以提供过滤：

```bash
ttmux swarm listen <群> --as web --mentions --once
```

含义：只输出 `@web`、`@all`、`#自己负责的卡`、以及 master 回复链相关的新消息。

#### Web 支撑

Web 广场输入框建议支持成员选择器：

- 输入 `@` 弹出 `master / human / all / 成员列表`。
- 从卡片详情发消息时自动带 `#<卡id>`。
- 从“追加需求 / 重新打开”入口发消息时自动带 `@master`。
- 展示消息时高亮 `@xx` 和 `#tN`，但不改变原始消息文本。

### 3.4 消费游标

在每个蜂群库或元数据目录里记录 listener 游标：

```
~/.local/share/ttmux/swarms/<群>/
└── listeners/
    ├── master.last_post
    ├── api.last_post
    └── web.last_post
```

游标语义：

- 文件内容是该 listener 已完成一轮理解和决策的最大 post id。
- 拉取命令：`ttmux swarm feed <群> --since "$(cat listeners/<name>.last_post)" --json`
- 处理成功后更新到本批最大 id。
- 若本轮 agent 崩溃，不更新游标，下轮会重读；agent 需要看上下文避免重复执行破坏性动作。

### 3.5 完成后的 reopen

蜂群 `status=done` 不代表 master 停止读取 human。完成后的监听策略：

1. master 进入低频监听（例如 60s）。
2. 如果出现 `author=human` 新消息：
   - 先读 `swarm status`、`swarm board`、最近 feed；
   - 判断是追问、验收反馈、bug、增量需求还是闲聊；
   - 需要继续工作时，把蜂群状态从 `done` 调回 `running` 或 `integrating`；
   - 新建 / 重开看板卡，派给现有或新成员；
   - 广场发 `kind=decide` 说明处理决定。

建议新增命令：

```bash
ttmux swarm reopen <群> [--reason "..."]
```

M1 可先由 master 直接用现有能力实现：新增卡片、移动卡片、继续派活；状态字段可后补。

## 4. Master Loop

### 4.1 每轮输入

master 每轮至少读取：

```bash
ttmux swarm feed <群> --since <last_post> --json
ttmux swarm board <群> --json
ttmux swarm status <群> --json
```

有以下信号时再兜底抓终端：

- 新消息是 `done` / `block` / `ask`；
- 看板有卡进入 `review` 或 `blocked`；
- human 要求检查某个成员产出；
- worker 长时间无进展。

### 4.2 决策优先级

1. **human 新指令**：无论蜂群是否 done，都先判断。
2. **审批 / 阻塞**：会卡住执行链，快速处理。
3. **review / done**：验收成员产出，必要时标 `swarm done <群> <成员>`。
4. **看板调度**：派新卡、重派、解锁依赖。
5. **普通进度**：更新记忆，低优先处理。

### 4.3 对 human 指令的处理模板

master 读到 human 消息后必须做四步：

1. **解释意图**：这是新需求、修改验收标准、追问状态、叫停、返工，还是无操作消息。
2. **对照当前状态**：读看板和成员完成状态，判断影响哪些卡 / 成员。
3. **执行或拒绝**：能执行则建卡 / 派活 / 发 worker 指令；不明确则在广场提问。
4. **回写广场**：用 `kind=decide` 或 `kind=ask` 明确告诉 human 已如何处理。

示例：

```bash
ttmux swarm say <群> --kind decide --re <human_post_id> \
  "收到：这是登录页的增量返工。我已重开卡 t4 并派给 web，验收标准补充为..."
```

## 5. Worker Loop

worker 不做全局调度，但也要读广场，避免“master 已裁决但 worker 没看到”。

### 5.1 每轮输入

```bash
ttmux swarm feed <群> --since <last_post> --json
ttmux swarm task ls <群> --assignee <自己> --json
```

如果新消息里引用了自己的卡，再读：

```bash
ttmux swarm task show <群> <卡id>
```

### 5.2 worker 必须响应的消息

- `author=master` 且 `kind=decide|broadcast`，并且文本点名自己、点名自己的卡、或是全员指令。
- `author=human` 且文本 `@自己`。
- 文本含 `@all`，但 worker 只判断自己是否需要动作，不做全局调度。
- 与自己负责卡片相关的 `block/ask/decide` 回复链。
- 自己卡片被移动到 `assigned` / `doing` / `review` 后的后续说明。

worker 响应时只做自己范围内的动作：

- 更新自己卡片状态；
- 继续实现 / 测试 / 返工；
- 用 `swarm say --kind note|ask|block|done` 汇报；
- 完成后执行 `ttmux swarm done <群> <成员>`。

## 6. CLI 与脚本落点

### 6.1 新增 listener helper

新增 `cli/ttmux-cli/lib/listener.sh`，提供小而通用的 helper：

```bash
ttmux swarm listen <群> --as <master|成员> [--once] [--interval 10]
```

M1 中 `listen` 不替 agent 做决策，只负责：

1. 读 `<as>.last_post`；
2. 拉 `feed --since --json`；
3. 解析 `@xx` / `#tN`，按 `--as` 过滤或标注相关性；
4. 输出一段适合 agent 阅读的“新广场消息 + 看板摘要 + 状态摘要”；
5. `--once` 模式返回后由 agent 自己判断；
6. 非 `--once` 模式可作为纯终端 watch，但不推荐代替 agent loop。

更务实的第一步是只加两个内部函数：

- `_listener_last_get <swarm> <name>`
- `_listener_last_set <swarm> <name> <post_id>`

然后把 loop 约束写进 master / worker prompt。

### 6.2 Prompt 落点

修改：

- `backend/api/prompts/master.md.tmpl`
- `backend/api/prompts/worker.md.tmpl`
- `skills/cc-swarm/docs/patrol.md`

master prompt 追加硬约束：

```text
你必须持续监听广场。每轮先读:
ttmux swarm feed {{.Swarm}} --since <你的last_post> --json
human 的新消息优先级最高。即使蜂群已完成，也要判断是否需要重开、派活或答复。
```

worker prompt 追加硬约束：

```text
你必须定期读广场，处理 @{{.Member}}、你负责的卡片、master 的 decide/broadcast。
不要给其他成员派活；需要全局调度时向 master 提问。
```

### 6.3 Web 落点

Web 看板 / 广场发 human 消息时，继续走：

```bash
ttmux swarm say <群> --as human --kind note|ask|block "@master <消息>"
```

建议 Web 端增加两个细节：

- human 从看板卡片详情发消息时自动带 `#<卡id>`。
- “重新打开 / 追加需求”入口默认发 `kind=ask` 或 `kind=note`，并自动带 `@master`。
- 输入 `@` 时弹出 master、all、human 和成员列表，选中后插入 `@xx`。

## 7. 状态与幂等

### 7.1 去重原则

- 游标只代表“已读并完成判断”，不是“动作一定成功”。
- 动作前先查当前状态，例如卡是否已经 done、成员是否已经 marked done。
- 对同一 human post 产生的决策消息必须带 `--re <post_id>`，方便审计。
- 对破坏性或外部副作用动作，仍需遵循现有审批规则。

### 7.2 重启恢复

agent 重启后：

1. 读自己的 last_post；
2. 读最近 20 条 feed 作上下文；
3. 读 board/status；
4. 从 `last_post` 之后继续处理。

如果 last_post 丢失，默认从最近 50 条开始，让 agent 自行总结并设置游标。

## 8. 实施计划

### M1：文档 + prompt 约束

- 更新 master / worker prompt，明确“必须监听广场”。
- 更新 `cc-swarm` patrol 文档，把 human 消息列为蜂群作用域最高优先级。
- 不改 schema，不引后台 daemon。

### M2：listener 游标

- 增加 `listeners/<name>.last_post` helper。
- 增加 `ttmux swarm listen <群> --as <name> --once`，输出新消息 + 摘要。
- `listen` 支持解析 `@xx` / `#tN`，给 master 输出全量，给 worker 标出与自己相关的消息。
- master / worker prompt 改为优先使用 `listen --once`。

### M3：完成后重开

- 增加 `ttmux swarm reopen <群> --reason`。
- master 处理 done 状态下的 human 消息时自动 reopen。
- Web 上展示“已完成但有新 human 消息待处理”的状态。

### M4：事件推送

- 后端对 `posts` 表提供 SSE / WebSocket 增量流。
- Web 端 human 发言后可显示“master 已读 / 已决策”的状态。

## 9. 验收标准

1. swarm 标记 done 后，human 在广场发“这里还要补 X”，master 下一轮会读到并做判断。
2. master 对 human post 产生的处理结果会以 `kind=decide|ask` 且 `--re <post_id>` 回写广场。
3. human 在 Web 广场输入 `@master` 后，master 下一轮必读；输入 `@web #t4` 后，web worker 能识别为相关消息。
4. worker 能看到 master 的返工 / 裁决消息，并只处理自己相关的卡。
5. 同一条 human 消息不会被无限重复派活。
6. claude 和 codex 成员遵循同一套监听规则。

## 10. 风险

- 纯 prompt 约束可能因 agent 忙于执行而漏读；M2 的 `listen --once` 能降低漏读成本。
- 轮询频率太高会增加 API 和 shell 调用成本；建议 master 10-30s，worker 30-60s，done 后 master 60s。
- human 消息语义可能不清楚；master 不应擅自扩大范围，不清楚就用 `kind=ask` 回问。
- 多个 master 同时存在会重复处理 human 消息；需要保持“一个蜂群一个 supervisor / master”的约束。
